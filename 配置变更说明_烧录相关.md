# P01 烧录相关配置变更说明

本文档说明此前为配合 Cursor 烧录而对 P01 项目做过的修改，便于你排查“在 Keil 里能成功烧录一次，之后未改代码仍无法烧录”的问题。

---

## 一、被修改过的文件与内容

### 1. `P01_flight_hal.uvprojx`（工程文件）

| 项 | 修改内容 | 可能影响 |
|----|----------|----------|
| **OutputDirectory** | 由 `P01_flight_hal\` 改为 `.\P01_flight_hal\` | 仅路径写法不同，在 Keil 里从工程目录打开时效果一致。若用命令行/其他工作目录打开，可能影响输出路径。 |
| **AfterMake → RunUserProg2** | 改为 `0`（关闭“编译后自动下载”） | 编译后不会自动烧录，需手动点 Download。 |
| **DllOption → TargetDllName** | 设为 `SARMCM3.DLL` | 调试/下载用的目标 DLL，与 P02 一致。 |
| **Utilities → Flash2** | 设为 `BIN\UL2V8M.DLL` | Flash 编程算法，与 P02 一致。 |

**未改动的：** 代码、源文件列表、芯片型号、Flash 算法名（STM32F10x_128）等均未改。

### 2. `P01_flight_hal.uvoptx`（选项/会话文件，**会被 Keil 在关闭时覆盖**）

| 项 | 说明 |
|----|------|
| **DebugOpt → pMon** | 调试/下载使用的 DLL，当前为 `STLink\ST-LINKIII-KEIL_SWO.dll`。 |
| **TargetDriverDllRegistry → ST-LINKIII-KEIL_SWO** | 里存有 ST-Link 的序列号、Flash 算法路径、连接参数等。 |

**重要：** 每次关闭 Keil，都会把当前界面里选的调试器、连接选项等写回 `uvoptx`。若某次在 Keil 里改过 Debug 设置、或连接失败后 Keil 写入了错误状态，下次打开就会用这份“坏”配置，导致时而能烧录、时而不能。

### 3. `.vscode/tasks.json`

- 新增任务 **「Flash (STM32CubeProgrammer)」**：用 STM32CubeProgrammer 命令行烧写 `P01_flight_hal.hex`，不经过 Keil 的 Download。
- 未改 Keil 工程本身，只增加 Cursor 里的一种烧录方式。

### 4. 其他（脚本/说明等）

- 若存在 `flash_with_stlink.bat`、`安装ST-Link_V2驱动.bat`、`ST-Link_V2驱动说明.txt` 等，均为本地辅助脚本或说明，**不参与 Keil 的编译与下载**，不会导致“在 Keil 里一次能烧一次不能”的现象。

## 二、“在 Keil 里成功烧录一次，之后没改代码仍无法烧录”的常见原因

1. **`uvoptx` 被覆盖**  
   关闭 Keil 时会把当前 Debug/ST-Link 设置写回 `uvoptx`。若当时连接异常或改过选项，下次打开就会用这份配置，导致再次烧录失败。  
   **建议：** 在 Keil 里能稳定烧录时，备份一份 `P01_flight_hal.uvoptx`，以后若又出现“莫名不能烧录”，用备份覆盖当前 `uvoptx` 再试。

2. **InvalidFlash 状态**  
   `uvprojx` 里有 `<InvalidFlash>1</InvalidFlash>` 时，表示 Keil 认为当前 Flash 配置无效，可能影响下载或后续选项。  
   **建议：** 在 Keil 中打开 **Options for Target → Debug → 使用 ST-LINK(III)**，再点 **Settings**，确认 Flash Download 里算法正确，然后确定退出；若 Keil 不再报错，有时会把 InvalidFlash 清掉。

3. **ST-Link 被占用**  
   STM32CubeProgrammer、Cursor 的烧录任务、其他 Keil 实例等若正在使用 ST-Link，会导致“ST-LINK USB communication error”。  
   **建议：** 烧录前关掉其它可能用 ST-Link 的程序，只保留一个 Keil 且只点一次 Download。

4. **工作目录/调用方式不同**  
   用 Cursor 的 Keil 助手或脚本调用 UV4 时，工作目录可能不是 `MDK-ARM`，和直接双击 `uvprojx` 在 Keil 里打开不同，可能影响相对路径或调试器加载。  
   **建议：** 若只在“直接打开 Keil 并点 Download”时能烧录，则优先在 Keil 里烧录；在 Cursor 里用「Flash (STM32CubeProgrammer)」任务烧写 hex 作为备用方案。

## 三、若想恢复“改前”的工程配置（仅 uvprojx）

若你希望工程配置尽量回到修改前，可在 **仅** `P01_flight_hal.uvprojx` 中做如下还原（用文本编辑器或 Keil 均可）：

- **OutputDirectory**：改回 `P01_flight_hal\`（若你更习惯原来的写法）。
- **AfterMake → RunUserProg2**：若以前是“编译后自动下载”，可改回 `1`，并填回原来的 **UserProg2Name**（若有）。
- **DllOption / Flash2**：若你之前用的是别的 DLL，再改回你原来的；若本来就和 P02 一样，可保持现状。

**注意：** `uvoptx` 没有“唯一正确版本”，会随 Keil 使用而变化，一般不做逐项还原，而是用“备份恢复”的方式。

## 四、推荐日常用法（避免配置被“带偏”）

1. **在 Keil 里烧录时**  
   - 用 Keil 打开 `P01_flight_hal.uvprojx`，编译后点 Download。  
   - 若某次能稳定烧录，复制一份 `P01_flight_hal.uvoptx` 存为 `P01_flight_hal.uvoptx.working` 备份。

2. **在 Cursor 里烧录时**  
   - 不依赖 Keil 的 Download：先编译（Keil 或 Cursor 的构建任务），再运行 **「Flash (STM32CubeProgrammer)」** 任务烧写 `P01_flight_hal.hex`，避免受 uvoptx/工作目录影响。

3. **再次出现“莫名不能烧录”时**  
   - 若有 `P01_flight_hal.uvoptx.working`，用其覆盖当前 `uvoptx` 再试。  
   - 检查是否有其它程序占用 ST-Link。  
   - 在 Keil 里重新选一次 Debug 里的 ST-LINK 并确认 Settings，让 Keil 刷新一次 Flash 配置。

以上即为当前项目中与烧录相关的配置变更及可能原因，便于你按需还原或排查。
